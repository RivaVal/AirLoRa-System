## 🛑 ЭТАП 1: ПОЛНАЯ И БЕЗОПАСНАЯ ОСТАНОВКА ПРОЕКТА
*Решает проблемы: `permission denied`, зависшие контейнеры, кэш Docker, рассинхронизация прав.*

Выполни в терминале **хост-машины** (не в контейнере):

```bash
# 1. Переход в папку проекта
cd ~/Y-2/Raspberry/ESP32_DevMode_AirPlane/AirLoRa-System/aircraft/drone_v31_video

# 2. Попытка стандартной остановки
sudo docker compose down -v --remove-orphans 2>/dev/null

# 3. Если контейнер зависает (ошибка permission denied / Error while Stopping):
#    Находим ID контейнера
CONTAINER_ID=$(sudo docker ps -q -f name=rpi-zero2w-dev)
if [ -n "$CONTAINER_ID" ]; then
    echo "⚠️ Контейнер зависает. Принудительное завершение..."
    sudo docker kill -s SIGKILL "$CONTAINER_ID" 2>/dev/null
    sudo docker rm -f "$CONTAINER_ID" 2>/dev/null
    sudo docker compose down -v --remove-orphans 2>/dev/null
fi

# 4. Очистка кэша, образов и томов (освобождает RAM/Disk)
sudo docker system prune -af --volumes

# 5. Проверка чистоты
docker ps -a
```
✅ **Ожидаемый результат:** Вывод пустой или содержит только `buildx_buildkit_multiarch0`. Все ресурсы освобождены.

> 💡 **Почему возникали ошибки раньше?**  
> `docker compose` без `sudo` не имеет прав на `/var/run/docker.sock`. Контейнер с `code-server`/`ffmpeg` мог заблокировать дескрипторы, из-за чего `SIGTERM` игнорировался. `SIGKILL` + `system prune` гарантированно сбрасывает состояние.

## 📜 ЭТАП 5: АРХИТЕКТУРА, АЛГОРИТМЫ И ДОКУМЕНТАЦИЯ

### 🔹 1. Захват кадров (`camera_interface.py`)
**Архитектура:** Паттерн Strategy. Базовый класс `CameraInterface` задаёт контракт. `MockCamera` и `RealCamera` наследуют его.
**Алгоритм:** 
- `MockCamera`: генерирует `numpy.zeros()`, рисует геометрические примитивы через `cv2.rectangle/circle`, добавляет таймер. Не требует GPU, работает на x86.
- `RealCamera`: обёртка над `picamera2`. При старте инициализирует ISP, настраивает `video_configuration`, возвращает `np.ndarray` в `BGR`.
**Почему так?** Позволяет отлаживать 100% логики пайплайна до получения железа. Переход на реальную камеру = замена 1 флага.

### 🔹 2. Приём телеметрии (`uart_telemetry.py`)
**Архитектура:** Многопоточный `threading.Thread` + `threading.Lock` для атомарного доступа к данным.
**Алгоритм:**
- Поток блокируется на `ser.readline()` с таймаутом 0.1с.
- Парсит формат `KEY1=VAL1;KEY2=VAL2`.
- Если UART недоступен → автоматически генерирует мок-данные (`ALT`, `BAT`, `GPS`, `MODE`).
- `get()` возвращает глубокую копию словаря, исключая race-condition при чтении из главного цикла.
**Почему так?** Чтение из последовательного порта блокирующее. Вынос в отдельный поток гарантирует стабильные 30 FPS видео без дропов кадров.

### 🔹 3. Наложение телеметрии (`camera_pipeline.py`)
**Архитектура:** Обработка "на лету" (online overlay) перед кодированием.
**Алгоритм:**
1. Получение кадра из камеры.
2. Вызов `telemetry.get()`.
3. `cv2.putText()` рисует текст в координатах `(10,30)` и `(480,30)`.
4. Запись байтов в `ffmpeg.stdin`.
**Сложность:** O(1) на кадр. На 640x480 занимает ~1.2 мс CPU, не влияет на FPS.

### 🔹 4. Кодирование и передача (`ffmpeg` pipe)
**Архитектура:** Дочерний процесс `subprocess.Popen` с перенаправлением `stdin`.
**Алгоритм:**
- Python пишет сырые `BGR`-байты в pipe.
- `ffmpeg` принимает поток: `-f rawvideo -s 640x480 -pix_fmt bgr24 -r 30`
- Кодирует в MJPEG: `-c:v mjpeg -q:v 5` (баланс качества/размера)
- Отправляет по UDP: `-f mjpeg udp://IP:PORT`
**Почему MJPEG+UDP?** 
- MJPEG: каждый кадр самодостаточен. Потеря UDP-пакета = 1 артефакт, а не рассинхрон всего потока.
- Pipe: исключает запись на диск, минимизирует задержку (<40 мс от захвата до отправки).

### 🔹 5. Наземная станция (`ground_station.py`)
**Архитектура:** Разделение потоков декодирования и вывода.
**Алгоритм:**
- При `--record`: `ffmpeg` слушает UDP, копирует поток без перекодирования (`-c:v copy`), пишет в `.mp4`. Нагрузка CPU ≈ 2%.
- При `--display`: `ffplay` с флагами `-fflags nobuffer -flags low_delay -framedrop` отключает буферизацию, отображает кадры сразу по приходу. Задержка ~60-90 мс.
**Почему два процесса?** `ffplay` оптимизирован под live-вывод, но плохо пишет файлы. `ffmpeg` надёжно архивирует, но не имеет GUI-рендера. Разделение даёт оба преимущества без конфликтов.

## 🔜 ПЕРЕХОД НА РЕАЛЬНОЕ ЖЕЛЕЗО (Pi Zero 2W)
1. Скопируй проект на плату: `scp -r drone_v31_video pi@<IP>:/home/pi/`
2. Установи драйверы: `sudo apt install python3-picamera2 python3-opencv ffmpeg`
3. Раскомментируй `devices:` в `docker-compose.yml`
4. Запусти на плате: `python3 camera_pipeline.py --telemetry --auto-hq 5 --udp udp://<GROUND_IP>:5000`
5. Камера и UART активируются автоматически. Mock-режим отключится.
